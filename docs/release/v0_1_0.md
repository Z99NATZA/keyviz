# v0.1.0 - Windows Key Visualizer

Release date: 2026-07-14

Initial KeyViz release for showing keyboard input in Windows coding videos and
screen recordings. The app captures keyboard input globally, renders recent
text and special keys as one ordered inline history, and stays visible without
taking focus or blocking interaction with the active application.

## Highlights

- Added global keyboard input capture through Windows Raw Input.
- Added a transparent, always-on-top, click-through WPF overlay.
- Added ordered inline rendering in the form `text [Special] text`.
- Added configurable text length and special-key history limits.
- Added left, center, and right bubble positions with even horizontal spacing.
- Added optional Show and Hide controls plus a System Tray menu.
- Added a dark-only UI based on the wfchat dark palette and control styling.
- Added self-contained `win-x64` publishing with no external NuGet dependency.

## Keyboard Input

- Registers for keyboard Raw Input with `RIDEV_INPUTSINK`, allowing KeyViz to
  receive input while another application has focus.
- Handles key-down and key-up events without blocking or modifying input sent
  to other applications.
- Normalizes left and right Shift, Ctrl, and Alt virtual keys.
- Tracks active modifiers and formats shortcuts such as `Ctrl + Shift + S`.
- Processes input in memory only and does not create a keystroke log.

## Inline Token Display

- Stores display history as ordered `Text` and `Special` tokens.
- Renders tokens on one line, for example:

  ```text
  hello [Space] world [Ctrl + S] next
  ```

- Appends consecutive printable keys to the current text token.
- Displays `Space`, `Shift`, `Ctrl`, `Alt`, `Win`, `Enter`, `Backspace`,
  navigation keys, function keys, and shortcuts as special-key chips.
- Replaces a modifier preview such as `[Ctrl]` with the completed shortcut such
  as `[Ctrl + S]` instead of counting both.
- Counts every special-key chip as one item regardless of label length.
- Applies Backspace to the latest text token and records `[Backspace]` in the
  ordered history.
- Scrolls horizontally to keep the newest token visible when the line exceeds
  the bubble width.
- Fades the display after three seconds of inactivity and starts a fresh group
  on the next input.

## Overlay And Controls

- Uses a borderless transparent WPF window with `WS_EX_TRANSPARENT`,
  `WS_EX_NOACTIVATE`, and `WS_EX_TOOLWINDOW`.
- Keeps the overlay above ordinary windows without taking focus or intercepting
  mouse input.
- Supports `left`, `center`, and `right` bubble positions centered at
  approximately 25%, 50%, and 75% of the primary work area.
- Keeps at least 32 pixels between the overlay window and the horizontal work
  area edges.
- Provides optional Show and Hide buttons in a separate non-activating control
  window at the lower-left corner.
- Provides Show, Hide, and Exit actions through the System Tray even when the
  on-screen controls are disabled.

## Settings

KeyViz creates and reads the following file at startup:

```text
%LocalAppData%\KeyViz\settings.json
```

Supported settings:

```json
{
  "maxTextLength": 20,
  "maxSpecialKeys": 3,
  "showControls": false,
  "bubblePosition": "center"
}
```

- `maxTextLength`: total text characters retained, clamped from 0 to 200.
- `maxSpecialKeys`: special tokens retained, clamped from 0 to 20; one chip
  always counts as one item.
- `showControls`: shows or hides the lower-left Show and Hide buttons.
- `bubblePosition`: accepts `left`, `center`, or `right` and falls back to
  `center` for unsupported values.
- Settings are loaded once at startup, so changes require restarting KeyViz.
- Invalid or unreadable JSON falls back to defaults without overwriting the
  existing file.

## Dark Theme

- Uses one dark theme with no light-mode toggle.
- Keeps shared semantic colors in `App.xaml` while component layout and sizing
  remain local to each XAML window.
- Uses the wfchat dark palette for app panels, nested soft surfaces, text,
  borders, actions, and shadows.
- Uses a softer inactive state for Show and Hide while preserving the current
  action color for the active state.

## Privacy And Runtime Behavior

- Keeps key state and display tokens in process memory only.
- Does not write typed content to files, logs, a database, or the network.
- Does not require administrator privileges for normal use.
- Uses a console-attached Debug build so `dotnet run` can stop cleanly with
  `Ctrl+C` while the terminal has focus.
- Uses a GUI Release build so the published application does not open a console
  window.

## Known Limitations

- Printable-character conversion currently follows an English keyboard layout
  and may not match Thai or other active keyboard layouts, dead keys, or input
  method editors.
- The overlay positions against the primary Windows work area; monitor
  selection is not configurable.
- Secure Desktop surfaces such as UAC prompts are not captured or overlaid.
- Exclusive-fullscreen games may appear above the overlay; borderless windowed
  mode is more reliable.
- Settings require manual JSON editing and an application restart; there is no
  settings window or live reload.
- Mouse buttons, Start with Windows, and a project-specific tray icon are not
  included in this release.
- Automated tests do not yet cover keyboard-to-token formatting end to end.

## Upgrade Notes

- This is the first release and does not require data migration.
- Existing `%LocalAppData%\KeyViz\settings.json` files remain compatible when a
  new field is absent because the application uses that field's default value.
- The application does not rewrite an existing settings file to add missing
  fields; copy the desired field from `settings.example.json` when needed.
- A self-contained publish includes the .NET runtime, so target Windows 10/11
  machines do not need a separate .NET installation.
